Minor Issues mm/userfaultfd: re-validate vma in mfill_atomic() loop under CONFIG_PER_VMA_LOCK Deepanshu Kartikey 1p
Adds a per-iteration VMA bounds check in mfill_atomic() to detect and recover from concurrent VMA splits under CONFIG_PER_VMA_LOCK, where a racing UFFDIO_UNREGISTER can shrink vma->vm_end past the current dst_addr via a race window in vma_start_read().

The comment opener uses a tab then seven spaces rather than a plain

> +		if (state.dst_addr < state.vma->vm_start ||
> +		    state.dst_addr >= state.vma->vm_end) {
> +			mfill_put_vma(&state);
> +			state.dst_start = state.dst_addr;

I verified the resource management: mfill_put_vma() guards against a
NULL vma, so the mfill_put_vma() at line 938 is a no-op when
mfill_get_vma() fails here.  No double-release.

I confirmed that state.dst_start and state.len are exactly what
mfill_get_vma() passes to uffd_mfill_lock().  Setting dst_start =
dst_addr and len = (dst_start + len - dst_addr) is correct:
uffd_mfill_lock() under CONFIG_PER_VMA_LOCK finds the VMA at the new
position and validates the remaining range fits within it.  If the
remaining range spans VMAs (e.g. multiple concurrent splits),
mfill_get_vma() returns -ENOENT, the loop breaks, and the caller sees
partial progress.  Userspace can retry with the outstanding range,
which is the existing contract.

I also checked mfill_copy_folio_retry(): it calls mfill_put_vma() and
mfill_get_vma() on the same state fields.  If this re-lookup already
fired on a prior iteration, the adjusted state.dst_start and state.len
are reused correctly by a subsequent copy retry, because dst_addr has
not advanced between the re-lookup and the retry.

One scope concern: the commit message and the comment both say "Under
CONFIG_PER_VMA_LOCK" but the new block is not wrapped in
#ifdef CONFIG_PER_VMA_LOCK.  Under the non-PER_VMA_LOCK path,
uffd_mfill_lock() holds mmap_read_lock() for the entire loop duration,
preventing concurrent VMA splits, so state.dst_addr >=
state.vma->vm_end can never be true there.  The check is dead code on
that path.  Would you consider wrapping the block in
#ifdef CONFIG_PER_VMA_LOCK ... #endif to match the stated intent and
avoid the per-iteration overhead on the mmap-lock path?

I traced the charge/uncharge symmetry, skip_next control flow, vmstat

One type issue is worth fixing before merging.

> diff --git a/mm/memcontrol.c b/mm/memcontrol.c
> --- a/mm/memcontrol.c
> +++ b/mm/memcontrol.c

[ ... ]

> +		struct pglist_data *pgdat;
> +		unsigned int batch_bytes;
> +		size_t run_len = 0;

batch_bytes is declared as unsigned int, but obj_size is size_t and
run_len is size_t.  The multiplication

    batch_bytes = obj_size * run_len;

is evaluated in size_t arithmetic and then silently truncated to
unsigned int on 64-bit builds.  For current callers the product stays
well below 4 GB, but the intermediate should be size_t to match
obj_size and the size_t parameter of __obj_cgroup_charge().
__consume_obj_stock() already takes unsigned int, so the narrowing
would happen at that call site instead of here.

> +		for (j = i + 1; j < size; j++) {
> +			struct slab *slab_j = virt_to_slab(p[j]);
> +			if (slab_pgdat(slab_j) != pgdat)
> +				break;
> +			if (!slab_obj_exts(slab_j) &&
> +			    alloc_slab_obj_exts(slab_j, s, flags, false)) {
> +				skip_next = true;
> +				break;
> +			}
> +			run_len++;
> +		}

I verified the skip_next advance logic:

    if (skip_next)
        i = i + run_len + 1;
    else
        i += run_len;

When alloc_slab_obj_exts() fails at position j, run_len++ has not
been reached (it follows the failure check), so run_len counts objects
i..j-1 only, and i advances to j+1.  The failed object at j is
skipped without any charge being issued, matching the original
continue behaviour.  Double-checked the two-consecutive-failure case:
the outer-loop guard handles the next skipped object independently via
its own i++; continue path.

> +		__account_obj_stock(objcg, stock, batch_bytes,
> +				    pgdat, cache_vmstat_idx(s));

The vmstat accounting is symmetric with __memcg_slab_free_hook(): all
objects in the batch share the same pgdat (enforced by the inner-loop
break), and the free hook unchargess per-object using slab_pgdat(slab)
for each, which resolves to the same pgdat.  Total charged bytes equal
obj_size * run_len; total bytes refilled across individual frees equal
the same.  The obj_cgroup_get() count in the assignment loop is one
per batch object, balanced by one obj_cgroup_put() per object in the
free hook.
