Needs Discussion mm/mempolicy: NUMA mempolicy mismatch during remote access 🔗 Lin Ruifeng 1p
RFC (no code change) reporting that SVA IOPF handling via a kworker ignores the faulting process's NUMA mempolicy because get_vma_policy() falls back to get_task_policy(current), where current is the kworker rather than the original process.
Lin Ruifeng <linruifeng4@huawei.com>

[RFC PATCH] mm/mempolicy: NUMA mempolicy mismatch during remote access

The bug is real.  I traced the call chain in the current tree to confirm:

  iommu_sva_handle_iopf() [kworker context]
    -> iommu_sva_handle_mm()
      -> handle_mm_fault(vma, addr, FAULT_FLAG_REMOTE, NULL)
        -> do_anonymous_page()
          -> alloc_anon_folio()
            -> vma_alloc_folio_noprof()
              -> get_vma_policy(vma, addr, order, &ilx)

In get_vma_policy() (mm/mempolicy.c):

> pol = __get_vma_policy(vma, addr, ilx);
> if (!pol)
>         pol = get_task_policy(current);

When no VMA-level policy is set, the fallback is get_task_policy(current),
which returns the kworker's mempolicy (NULL -> system default), not the
original process's mempolicy.  The FAULT_FLAG_REMOTE flag passed to
handle_mm_fault() is only checked by arch_vma_access_permitted(); it
does not influence mempolicy resolution anywhere in the allocation path.

This same issue affects mm/hmm.c, which also drives handle_mm_fault()
with FAULT_FLAG_REMOTE from a non-owner context.

A few design questions worth settling before writing a fix:

1. The mm->owner field tracks the canonical task for the mm_struct.
   One approach is to make get_vma_policy() accept a task_struct*
   argument (or a mempolicy* fallback) so that FAULT_FLAG_REMOTE
   callers can supply the owner's policy.  However, mm->owner is
   maintained under alloc_lock and can change; any dereference needs
   RCU read-side protection.  What locking would the caller hold at
   the point of alloc_anon_folio() when invoked from a kworker?

2. Alternatively, could the policy be snapshotted from the owner task
   at bind time (iommu_sva_bind_device) and stored on the SVA domain?
   That avoids the per-fault owner lookup but means the policy is
   frozen at bind time and won't track subsequent set_mempolicy() calls.

3. The iommufd path (iommufd_fault_iopf_handler -> userspace response
   loop) delegates fault handling to userspace entirely and is not
   affected.  The SVA kernel handler iommu_sva_handle_iopf() is the
   affected path.

No code is changed in this RFC, so there are no correctness or locking
issues to flag yet.  Looking forward to the proposed fix.

--- [RFC PATCH 1/4] mm/numa: introduce nearest_nodes_nodemask() ---

> diff --git a/include/linux/numa.h b/include/linux/numa.h
> +static inline int nearest_nodes_nodemask(int node,
> +					 const nodemask_t *mask,
> +					 nodemask_t *out)
> +{
> +	return 0;
> +}

The no-NUMA stub returns 0 (success) without touching *out at all,
leaving it uninitialized. The kdoc for the real implementation says
"Return: 0 on success" and that *out is populated on success. Any
caller that does:

  if (!nearest_nodes_nodemask(n, &mask, &out))
      /* use out */

will read garbage on a CONFIG_NUMA=n build. The stub should clear
*out before returning, matching what the real implementation does at
the top of the function.

--- [RFC PATCH 2/4] mm/memory-tiers: introduce socket-aware topology
                    management ---

> +static inline int mp_next_demotion_node(int nid)
> +{
> +	return 0;
> +}

> +{

The same issue applies to the no-CONFIG_MIGRATION stubs inside the

Additionally, the cover letter notes that mp_next_demotion_nodemask(),

> +static int policy_resolve_package_nodes(struct mempolicy *policy,
> +					nodemask_t *mask)
> +{
> +	unsigned int node, ret = 0;
> +	nodemask_t package_mask;

> +	if (!policy || !mask)

> +		if (nodes_empty(*mask)) {
> +			node = first_node(policy->nodes);
> +			ret = mp_get_package_nodes(node, &package_mask);
> +			if (ret)
> +				goto out;
> +			nodes_and(*mask, package_mask, policy->nodes);
> +			if (nodes_empty(*mask))
> +				goto out;
> +		}
> +	}
> +
> +out:
> +	return ret;
> +}

There is a correctness bug in the second fallback path. When the
second nodes_and() yields an empty *mask, the function jumps to out
with ret == 0 (the successful return left by the second
mp_get_package_nodes() call). The caller treats ret == 0 as success:

  if (policy_resolve_package_nodes(policy, &mask))
      mask = policy->nodes;

So it proceeds with an empty mask. In weighted_interleave_nodes(),
with il_weight exhausted or node not in the empty mask,
next_node_in() on an empty nodemask returns MAX_NUMNODES, which then
propagates up the call stack even though policy->nodes is fully
populated. The `if (nodes_empty(*mask)) goto out` in the fallback
must set a non-zero ret before jumping, so the caller falls back to
policy->nodes as intended.

> +static unsigned int weighted_interleave_nodes(struct mempolicy *policy)
> +{
>  	unsigned int node;
>  	unsigned int cpuset_mems_cookie;
> +	nodemask_t mask;
>  
>  retry:
>  	cpuset_mems_cookie = read_mems_allowed_begin();
>  	node = current->il_prev;
> +
> +	if (policy_resolve_package_nodes(policy, &mask))
> +		mask = policy->nodes;
> +
> +	if (!current->il_weight || !node_isset(node, mask)) {
> +		node = next_node_in(node, mask);
>  		if (read_mems_allowed_retry(cpuset_mems_cookie))
>  			goto retry;

policy_resolve_package_nodes() is called unconditionally on every
entry to weighted_interleave_nodes(), including the common fast path
where il_weight is still positive and no new node selection is needed.
This means mp_get_package_nodes() runs for every single page
allocation under MPOL_WEIGHTED_INTERLEAVE.

What is the locking discipline inside mp_get_package_nodes()? The
implementation is in patch 2/4 which is truncated, but this matters:
weighted_interleave_nodes() is reached from mempolicy_slab_node(),
which can be called in GFP_ATOMIC or interrupt-disabled contexts. If
mp_get_package_nodes() acquires a mutex or sleeps, this is broken for
those callers.

The call should also be guarded so it is only made when a new node
actually needs to be selected (i.e., inside the il_weight == 0 branch),
both for correctness in atomic contexts and to avoid unnecessary work
on the hot path.

> +static unsigned int read_once_policy_package_nodemask(
> +					struct mempolicy *pol,
> +					nodemask_t *mask)
> +{
> +	nodemask_t package_mask;
> +
> +	barrier();
> +	if (policy_resolve_package_nodes(pol, &package_mask))
> +		memcpy(mask, &pol->nodes, sizeof(nodemask_t));
> +	else
> +		memcpy(mask, &package_mask, sizeof(nodemask_t));
> +	barrier();
> +
> +	return nodes_weight(*mask);
> +}

The barrier() idiom in read_once_policy_nodemask() works because it
wraps a plain memcpy of pol->nodes: the barriers prevent the compiler
from reordering memory accesses around that snapshot. Placing
policy_resolve_package_nodes() -- which may acquire locks and walk
data structures -- between two compiler barriers does not provide the
same guarantee and is misleading. If the intent is to take a stable
snapshot of the filtered nodemask for the weighted_interleave_nid()
path, the approach needs to be reconsidered.

This patch is not fully visible, but the title and the fs/binfmt_elf.c
change warrant attention.  The current call site in binfmt_elf.c is:

  /* If the header is requesting these pages to be
   * executable, honour that (ppc32 needs this).  */

parameter is removed?  Does the binfmt_elf.c change substitute a
