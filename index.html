Docs/mm/damon: document exclusivity of special-purpose modules

The Sphinx syntax fix and the cross-reference plumbing look correct.
One accuracy concern follows.

> diff --git a/Documentation/mm/damon/design.rst b/Documentation/mm/damon/design.rst

[ ... ]

> +.. _damon_design_special_purpose_modules_exclusivity:
> +
> +Note that these modules currently run in an exclusive manner.  If one of those
> +is already running, others will return ``-EBUSY`` upon start requests.

The phrase "if one of those is already running" implies mutual exclusivity
only among the three special-purpose modules (lru_sort, reclaim, stat).
But the actual check in damon_start() is:

    if ((exclusive && nr_running_ctxs) || ...)
        return -EBUSY;

nr_running_ctxs counts *all* active DAMON contexts, including any
user-space DAMON session started via the sysfs interface.  So a user
running their own DAMON monitoring through sysfs will also prevent any
of these modules from starting, and likewise an exclusive module already
running will block subsequent non-exclusive sysfs starts
(running_exclusive_ctxs path).

The same imprecision appears in the three admin-guide files:

> +Note that this module (damon_lru_sort) cannot run simultaneously with other
> +DAMON-based special-purpose modules.

This understates the actual restriction.  A user might read this as
"safe to enable alongside my own DAMON sysfs session", which is wrong.

Suggested wording for design.rst:

    Note that these modules request exclusive DAMON access on start.
    If any DAMON context is already running (whether started by another
    special-purpose module or via the sysfs interface), the start
    request will return ``-EBUSY``.

And for the three admin-guide files, something like:

    Note that this module cannot run simultaneously with any other active
    DAMON context, including other special-purpose modules and user-space
    DAMON sessions started via the sysfs interface.

Also a minor nit on the commit message: it says the cross-references are
added "at the end of their respective Example sections", but damon_stat
has no Examples section -- the note is placed inside the parameter
description block.  The placement looks reasonable; only the commit
message description is slightly off.

This isn't quite right for mounts with a size= limit.  When max_blocks
is set, shmem_inode_acct_blocks() enforces it and allocations fail with
EINOSPC well before memory is exhausted.  The next section covers this,
but the sentence here contradicts it.  Something like "by available
memory, swap, and any configured mount limit" would be more accurate.

> +Each tmpfs inode tracks two key counters: allocated pages (resident in
> +memory) and swapped pages (evicted to swap).  These are maintained by
> +the ``shmem_charge()`` and ``shmem_uncharge()`` accounting functions,
> +which keep the inode's block usage consistent with the filesystem's mount

the alloced counter and the used_blocks limit.  shmem_uncharge() calls
shmem_recalc_inode(inode, 0, 0), which updates neither counter directly.

shmem_recalc_inode() that pass a non-zero swapped argument (e.g.
shmem_writeout paths), not by shmem_charge/shmem_uncharge.  Attributing
both counters to these two functions is misleading.  Could this be
rephrased to describe what shmem_charge() and shmem_uncharge() actually
do (enforce block limits and trigger freed-page recalculation), and
separately note that shmem_recalc_inode() reconciles alloced vs. swapped
to compute how many pages have been freed?

Docs/mm: document Swap

Thanks for filling in this stub.  A few things need correction before
this is accurate.

> +Cluster Allocation
> +------------------
> +
> +Swap space is allocated in clusters (groups of contiguous slots, typically
> +32 pages).

This is wrong.  SWAPFILE_CLUSTER is defined in mm/swap.h as:

    #ifdef CONFIG_THP_SWAP
    #define SWAPFILE_CLUSTER    HPAGE_PMD_NR
    #else
    #define SWAPFILE_CLUSTER    256
    #endif

With CONFIG_THP_SWAP (the common case on x86_64) each cluster is 512
slots; without it, 256.  The value 32 does not appear anywhere in the
swap cluster code.

> +The swap cache is implemented as a per-cluster array of pointers
> +(the "swap table"), providing O(1) lookup by swap entry.
> +See also Documentation/mm/swap-table.rst.

Two issues here.

First, the swap table entries (atomic_long_t in struct swap_table) are
not plain pointers.  Each entry encodes one of four states via type
bits: NULL (free slot), Shadow (swapped-out, XA-compatible value plus
refcount), PFN (cached folio PFN plus refcount), or Bad.  The refcount
for copy-on-write sharing is packed into the same word.  Describing
them as "pointers" loses the most interesting part of the design.

Second, Documentation/mm/swap-table.rst does not exist.  The
implementation lives in mm/swap_table.h (an internal header).  Either
drop the cross-reference or create that document first.

Docs/mm: document Virtually Contiguous Memory Allocation

Thanks for filling in this stub.  A few technical points need
correction against the current implementation in mm/vmalloc.c.

> +Virtual Address Management
> +--------------------------
> +
> +The kernel reserves a large region of virtual address space for vmalloc
> +(on x86-64 this is hundreds of terabytes).  Within this region, allocated
> +and free ranges are tracked by ``struct vmap_area`` nodes organized in two
> +red-black trees — one sorted by address for the busy areas, and one
> +augmented with subtree maximum gap size for the free areas.  The augmented
> +tree allows free-space searches in O(log n) time.

The description of the busy-area tree is inaccurate.  Since the
vmap_node sharding work the busy areas are kept in *per-node*
red-black trees (vn->busy.root inside each struct vmap_node),
not a single global tree.  Only the free-area side remains a
single global augmented rbtree (free_vmap_area_root).  The text
should reflect this N-way sharding and explain why it was done
(to reduce lock contention on vmap_area_root).

> +Guard Pages
> +-----------
> +
> +By default, each vmalloc area is surrounded by a guard page — an unmapped
> +page that causes an immediate fault if code overruns the allocation.

"Surrounded" implies pages on both sides.  The implementation
adds only one guard page (size += PAGE_SIZE in __get_vm_area_node)
at the high end of the allocation.  Adjacent allocations share
that boundary, so there is no separate lower guard page.  Could
you say "followed by a guard page" or "terminated by a guard
page" instead?

> +Huge Page Support
> +-----------------
> +
> +On architectures that support it, vmalloc can use PMD- or PUD-level
> +mappings instead of individual PTEs, reducing TLB pressure for large
> +allocations.

P4D-level mappings are also supported.  vmap_try_huge_p4d() is
called from vmap_pud_range() and guarded by arch_vmap_p4d_supported().
The list should read "PMD-, PUD-, or P4D-level mappings".

> +Freeing
> +=======
> +
> +``vfree()`` can be called from any context, including interrupt handlers.

This is incorrect.  vfree() explicitly forbids NMI context:

    BUG_ON(in_nmi());

The kernel-doc comment for vfree() says:

    "Must not be called in NMI context"

Please correct this to: vfree() can be called from most contexts,
including interrupt handlers, but must not be called from NMI
context.  vfree_atomic() carries the same NMI restriction.

Good overview of the page allocator subsystem.  Two factual issues
below worth fixing.

> diff --git a/Documentation/mm/page_allocation.rst b/Documentation/mm/page_allocation.rst

[ ... ]

> +Per-CPU Pagesets
> +================
> +
> +Most order-0 allocations are served from per-CPU page lists (PCP) rather
> +than the global ``free_area``.  This avoids taking the zone lock on the
> +common path, which is critical for scalability on large systems.
> +
> +Each CPU maintains lists of free pages grouped by migratetype.  Pages are
> +moved between the per-CPU lists and the buddy in batches.  The batch size
> +and high watermark for each per-CPU list are tuned based on zone size and
> +the number of CPUs.
> +
> +When a per-CPU list is empty, a batch of pages is taken from the buddy.
> +When it exceeds its high watermark, excess pages are returned.
> +``lru_add_drain()`` and ``drain_all_pages()`` flush per-CPU lists when
> +the system needs an accurate count of free pages, such as during memory
> +hot-remove.

lru_add_drain() and drain_all_pages() are different per-CPU caches.
drain_all_pages() drains the PCP free-page lists (per_cpu_pages).  But
lru_add_drain() drains the LRU pagevec batches (struct cpu_fbatches in
mm/swap.c) -- pending LRU list additions that have not yet been
committed to the LRU proper.  These are unrelated to the PCP free-page
lists described in this section.  Could you either remove lru_add_drain()
from this paragraph, or add a sentence clarifying that it drains a
different per-CPU cache (pending LRU additions) rather than PCP pages?

[ ... ]

> +Memory compaction (``mm/compaction.c``) creates contiguous free blocks for
> +high-order allocations by relocating movable pages.  It runs two scanners
> +across a zone: one walks from the bottom to find movable in-use pages, the
> +other walks from the top to find free pages.  Movable pages are migrated
> +to the free locations, consolidating free space in the middle.

The description of the two scanners is correct, but the final phrase
looks wrong.  The migration scanner starts at the low PFN (bottom) and
walks upward; the free scanner starts at the high PFN (top) and walks
downward.  Movable pages found near the bottom are migrated to the free
slots found near the top.  After compaction, the region formerly
occupied by the migrated pages (the lower portion of the zone) is now
free, while the upper portion is now populated.  Free space ends up at
the bottom of the zone, not in the middle.  The scanners meet in the
middle, but that is the termination condition, not where the free space
concentrates.  Something like "leaving a large contiguous free region at
the lower end of the zone" would be more accurate.

Docs/mm: document Boot Memory

Good addition overall.  One factual error in the deferred struct-page
initialization section; a few smaller nits below.

> diff --git a/Documentation/mm/bootmem.rst b/Documentation/mm/bootmem.rst

[ ... ]

> +When regions are added, overlapping ranges are merged automatically.
> +Internally, ``memblock_add_range()`` handles insertion, overlap detection,
> +and merging in a single pass.  If the region array is full, it is doubled
> +in size — using memblock itself to allocate the new array.

memblock_double_array() checks slab_is_available() first.  When slab is
already up (which can happen during late boot region additions) it calls
kmalloc() instead.  Worth noting that the fallback is memblock only
during early boot before slab is ready.

[ ... ]

> +With ``CONFIG_DEFERRED_STRUCT_PAGE_INIT``, only pages in the boot node's
> +lower zones are initialized during early boot — enough to get the system
> +running.  The remaining pages are initialized in parallel by worker threads
> +(via the padata framework) before they are first needed.

This description has two inaccuracies worth fixing.

First, the scope is not limited to the "boot node".  defer_init() in
mm/mm_init.c applies the same logic to every NUMA node: lower zones of
all nodes are always initialized eagerly (the comment at line 728 reads
"Always populate low zones for address-constrained allocations"), while
only the highest zone on each node is subject to deferral.

Second, deferred pages are not guaranteed to be initialized "before they
are first needed".  The kthread running deferred_init_memmap() works
through the highest zone via padata, but deferred_grow_zone() also
exists precisely to handle on-demand initialization when an allocation
arrives before the background thread has reached that page-frame range.
So pages may be initialized either in the background or on first demand,
whichever comes first.

A more accurate description: with CONFIG_DEFERRED_STRUCT_PAGE_INIT,
all zones except the highest on each node, plus an initial section of
that highest zone, are initialized synchronously during boot.  The
remaining pages in each node's highest zone are then initialized in
parallel by per-node kthreads using the padata framework, or on demand
via deferred_grow_zone() if an allocation arrives first.

[ ... ]

> +Named Reservations
> +------------------
> +
> +The ``reserve_mem`` kernel command line parameter allows firmware or boot
> +loaders to reserve named memory regions that persist across kexec.

The persistence across kexec is conditional on CONFIG_KEXEC_HANDOVER
(via reserve_mem_kho_revive()).  Without that option the reservations
still work fine for their primary purpose — providing named physical
regions that drivers or subsystems can look up with
reserve_mem_find_by_name() — but they do not survive a kexec.  The
current wording could mislead readers into expecting cross-kexec
behavior unconditionally.

mm/vmalloc: free unused pages on vrealloc() shrink

Overall the direction is correct and the grow-in-place fix is clearly
necessary.  A few things worth checking:

> diff --git a/mm/vmalloc.c b/mm/vmalloc.c
> --- a/mm/vmalloc.c
> +++ b/mm/vmalloc.c

[ ... ]

> +		/* Free tail pages when shrink crosses a page boundary. */
> +		if (new_nr_pages < vm->nr_pages && !vm_area_page_order(vm)) {
> +			unsigned long addr = (unsigned long)p;
> +
> +			vunmap_range(addr + (new_nr_pages << PAGE_SHIFT),
> +				     addr + (vm->nr_pages << PAGE_SHIFT));
> +
> +			vm_area_free_pages(vm, new_nr_pages, vm->nr_pages);
> +			vm->nr_pages = new_nr_pages;
> +		}

vfree() checks (vm->flags & VM_FLUSH_RESET_PERMS) and, if set, calls
vm_reset_perms() which iterates over vm->nr_pages pages to reset their
direct-map permissions before releasing them to the buddy allocator.
After the partial free updates vm->nr_pages = new_nr_pages, a subsequent
vfree() will call vm_reset_perms() only on the surviving pages; the
tail pages freed here would have been released to the buddy allocator
without having their direct-map permissions restored.

Does vm_area_free_pages() (introduced in patch 1/3) call
set_area_direct_map() or otherwise reset direct-map permissions for
the freed pages when VM_FLUSH_RESET_PERMS is set?  Or should there be
a guard here:

    if (vm->flags & VM_FLUSH_RESET_PERMS)
        goto skip_partial_free;

to skip the optimisation for allocations that need the direct-map
reset path in vfree()?

Similarly, vfree() calls mod_lruvec_page_state(page, NR_VMALLOC, -1)
for each freed page (unless VM_MAP_PUT_PAGES is set).  Can you confirm
that vm_area_free_pages() mirrors that accounting for the freed tail
pages, so NR_VMALLOC stays accurate?

> +		vm->nr_pages = new_nr_pages;

After this update vm->pages[new_nr_pages..old_nr_pages-1] still hold
the (now-freed) page pointers.  vfree() won't touch them because it
stops at vm->nr_pages, so there is no double-free, but should
vm_area_free_pages() NULL those slots out so that any future debugging
visit (e.g. /proc/vmallocinfo or a crash dump) does not see stale
pointers?

[ ... ]

> -	if (size <= alloced_size) {
> +	if (size <= (size_t)vm->nr_pages << PAGE_SHIFT) {

The grow-in-place fix is correct.  alloced_size captures
get_vm_area_size(vm) once at entry; after a shrink reduces
vm->nr_pages, the old comparison would have allowed a grow into
already-freed pages.  Comparing against the live vm->nr_pages value
prevents that.
