mm/mempolicy: NUMA mempolicy mismatch during remote access

This RFC has no code diff attached.  The description is a problem report
asking for suggestions.  I traced the relevant code paths below.

The diagnosis is correct.  The call chain for SVA IOPF is:

  iommu_sva_handle_iopf()        [kworker context]
    -> iommu_sva_handle_mm()
         -> handle_mm_fault(..., FAULT_FLAG_REMOTE, ...)
              -> do_anonymous_page()
                   -> alloc_anon_folio()
                        -> vma_alloc_folio()
                             -> vma_alloc_folio_noprof()
                                  -> get_vma_policy(vma, addr, order,
                                                    &ilx)

In get_vma_policy():

> struct mempolicy *get_vma_policy(struct vm_area_struct *vma, ...)
> {
>     pol = __get_vma_policy(vma, addr, ilx);  /* vma->vm_policy */
>     if (!pol)
>         pol = get_task_policy(current);      /* falls back to kworker */

When there is no VMA-level policy (mbind() was not called, only
set_mempolicy() was used by the user process), get_task_policy(current)
returns the kworker thread's mempolicy, not the original task's.
FAULT_FLAG_REMOTE is set at the call site in iommu_sva_handle_mm()
but get_vma_policy() has no visibility into it.

There is a second related issue: vma_alloc_folio_noprof() also passes
numa_node_id() as the preferred nid to folio_alloc_mpol_noprof(), which
is the NUMA node of the kworker, not the faulting process.

Some thoughts on possible fix directions:

  1. mm->owner (struct task_struct __rcu *) is available from the
     vm_area_struct.  A remote-fault-aware path could read
     rcu_dereference(vma->vm_mm->owner)->mempolicy in place of
     current->mempolicy.  mm->owner can be NULL (all threads of a
     group may have exited while the mm is still alive) so a NULL
     check and appropriate fallback is required.

  2. A more general approach: add a task_struct * or mempolicy *
     field to struct vm_fault, populated when FAULT_FLAG_REMOTE is
     set, and thread it through to get_vma_policy() and
     vma_alloc_folio_noprof() so the nid hint is also correct.

  3. Bind-time snapshot: capture the mempolicy into the iopf_group
     or SVA domain at iommu_sva_bind_device() time and use it for
     all subsequent fault handling in that domain.

The problem description is accurate.  It would help to include a
reproducer and a proposed fix in the next revision.

comment explaining why it is safe to remove it (e.g. the new path is

bisectors in the future.

I traced the race through vma_start_read() in mm/mmap_lock.c.
The first seqcount check (READ_ONCE(vma->vm_lock_seq) ==
READ_ONCE(mm->mm_lock_seq.sequence)) passes, vm_refcnt is
incremented, then __vma_start_write() runs concurrently: it sets
WRITE_ONCE(vma->vm_lock_seq, mm_lock_seq) and shrinks vma->vm_end
via __split_vma(). After mmap_write_unlock() increments mm_lock_seq,
the second seqcount check in vma_start_read() sees vm_lock_seq !=
mm_lock_seq (they diverged), so vma_start_read() returns success
with a stale, already-split VMA. The bug is real.

 - Error path: kmem_cache_free_bulk() uncharges only objects whose
   obj_ext->objcg was assigned, so objects in the current batch (not
   yet assigned when __obj_cgroup_charge() fails) are correctly left
   uncharged.  Objects skipped due to alloc_slab_obj_exts() failure
   have obj_exts == 0, so memcg_slab_free_hook() returns early and
   never touches their obj_exts -- same as before.

 - get/put_slab_obj_exts() are not refcounted (they call
   metadata_access_enable/disable()); calling them once per object in
   the inner assignment loop is fine.

 - skip_next advancement: i = i + run_len + 1 correctly skips the one
   object whose alloc_slab_obj_exts() failed in the lookahead loop.

 - Stock paths (__refill_obj_stock sets cached_objcg before
   __account_obj_stock is called) satisfy the "call after
   __refill_obj_stock" precondition in both the stock-hit and
   stock-miss branches.

One type issue:

> diff --git a/mm/memcontrol.c b/mm/memcontrol.c
> --- a/mm/memcontrol.c
> +++ b/mm/memcontrol.c

[ ... ]

> +		unsigned int batch_bytes;
> +		size_t run_len = 0;

> +		batch_bytes = obj_size * run_len;

batch_bytes is unsigned int, but obj_size is size_t and run_len is
size_t, so obj_size * run_len is evaluated as size_t and then truncated
to 32 bits.  On a 64-bit kernel any batch where obj_size * run_len >=
4 GB wraps silently, causing memcg to be undercharged for the whole
run.  This is the same truncation that already exists in the
single-object path where obj_size is passed to __consume_obj_stock()
(which also takes unsigned int), but multiplication makes it reachable
at much smaller per-object sizes.

The truncated value then propagates to:

> +		if (!stock || !__consume_obj_stock(objcg, stock, batch_bytes)) {
> +			if (__obj_cgroup_charge(objcg, flags, batch_bytes, &remainder))

to 64 bits, but the value is already wrong.

> +		__account_obj_stock(objcg, stock, batch_bytes,
> +				    pgdat, cache_vmstat_idx(s));

__account_obj_stock() takes int nr.  If the truncated unsigned int
value exceeds INT_MAX, the implicit conversion makes nr negative,
decrementing NR_SLAB_RECLAIMABLE_B or NR_SLAB_UNRECLAIMABLE_B rather
than incrementing it.

Should batch_bytes be declared as size_t instead of unsigned int?

   maximum_alignment()=ELF_MIN_ALIGN=PAGE_SIZE, so the fast path is
   taken. After this patch alignment is always 64K > ELF_MIN_ALIGN,
   so every static-PIE exec on arm64/4K goes through an extra
   elf_load() probe and vm_munmap() even when the segments are only
   4K-aligned and contpte cannot fire. That is two extra syscalls on
   every exec for static-PIE binaries (ld.so invoked directly,
   static-pie applications) on arm64 with 4K pages.

2. ASLR entropy reduced for binaries that cannot benefit from contpte.

   For PIE (ET_DYN with PT_INTERP) the masking is:

     load_bias = ELF_ET_DYN_BASE + arch_mmap_rnd();
     if (alignment)
         load_bias &= ~(alignment - 1);

   A 64K-aligned load_bias only produces a 64K-aligned VMA if the
   segment vaddr is itself 0 mod 64K. A binary with p_align=4K can
   have a vaddr of e.g. 0x1000, so the mapped VMA starts at
   load_bias+0x1000 which is not 64K-aligned even after masking.
   Contpte does not fire for that mapping, yet exec_folio_order()
   bits of ASLR entropy are consumed on every exec of that binary.

Both problems would go away if the alignment bump were conditional on
the segments already being exec-folio-order aligned (vaddr, file
offset, and memsz all 64K-aligned), as WANG Rui suggests. Binaries
linked with p_align>=64K get the benefit; everything else keeps full
ASLR entropy and the static-PIE fast path.

One way to do this inside maximum_alignment() would be to only apply
the exec_folio_order floor when at least one PT_LOAD has
p_align >= (PAGE_SIZE << exec_folio_order()), so the ELF_MIN_ALIGN
guard in the static-PIE path still saves the fast path for
small-alignment binaries.

trade-off worth calling out in the commit message)?
