--- [PATCH v2 01/23] mm/vma: add vma_flags_empty(), vma_flags_and(),
    vma_flags_diff_pair() ---

> diff --git a/include/linux/mm_types.h b/include/linux/mm_types.h
> +/* Are no flags set in the specified VMA flags? */
> +static __always_inline bool vma_flags_empty(vma_flags_t *flags)
> +{
> +	unsigned long *bitmap = flags->__vma_flags;
> +
> +	return bitmap_empty(bitmap, NUM_VMA_FLAG_BITS);
> +}

Every other pure-query function in mm_types.h and mm.h takes a const
pointer: vma_flags_test(), vma_flags_test_any_mask(), and
vma_flags_test_all_mask() all declare "const vma_flags_t *".
vma_flags_empty() only reads the bitmap, so should the signature be:

  bool vma_flags_empty(const vma_flags_t *flags)

with a matching "const unsigned long *bitmap"?  As written, callers
that hold a const-qualified vma_flags_t cannot pass it without a cast.

> diff --git a/include/linux/mm.h b/include/linux/mm.h
> +static __always_inline vma_flags_t vma_flags_and_mask(
> +		const vma_flags_t *flags, vma_flags_t to_and)

> +	vma_flags_t dst;
> +	unsigned long *bitmap_dst = dst.__vma_flags;
> +	const unsigned long *bitmap = flags->__vma_flags;
> +	const unsigned long *bitmap_to_and = to_and.__vma_flags;
> +
> +	bitmap_and(bitmap_dst, bitmap, bitmap_to_and, NUM_VMA_FLAG_BITS);
> +	return dst;
> +}

Checked: with NUM_VMA_FLAG_BITS == BITS_PER_LONG this reduces to a
single-word AND.  bitmap_and() handles that correctly.

Checked the layout: vm_flags_t is "unsigned long"; vma_flags_t is

The hunk context confirms these additions land *inside* the body of
softleaf_entry_wait_on_locked(), between spin_unlock(ptl) and its
closing brace.  Defining a static inline function inside another
function body is not valid C (GCC nested-function syntax is not
used here), and the closing brace that was softleaf_entry_wait_on_locked's
own brace now becomes hmm_select_migrate's closing brace, leaving
softleaf_entry_wait_on_locked itself unclosed.  Any config that enables
HMM_MIRROR without CONFIG_MIGRATION will hit a compile error;
HMM_MIRROR depends only on MMU, so that combination is reachable
independently of the DEVICE_MIGRATION select added in patch 1.

The stub needs to appear after the closing brace of
softleaf_entry_wait_on_locked, not before it.

--- [PATCH v6 4/6] mm: setup device page migration in HMM pagewalk ---

> diff --git a/include/linux/migrate.h b/include/linux/migrate.h
> @@ -200,10 +201,14 @@ struct migrate_vma {
>  static inline enum migrate_vma_info hmm_select_migrate(struct hmm_range *range)

> -	return 0;

If HMM_PFN_REQ_MIGRATE is set in default_flags but range->migrate is

--- [PATCH v6 5/6] mm: add new testcase for the migrate on fault case ---

> +static int dmirror_fault_and_migrate_to_device(struct dmirror *dmirror,
> +					       struct hmm_dmirror_cmd *cmd)
> +{
> +	...
> +	struct migrate_vma migrate = { 0 };
> +	struct hmm_range range = { 0 };
> +	...
> +	migrate.pgmap_owner = dmirror->mdevice;
> +	migrate.src = src_pfns;
> +	migrate.dst = dst_pfns;
> +	migrate.flags = MIGRATE_VMA_SELECT_SYSTEM;

Neither migrate.start nor migrate.end is ever written.  Both remain 0.

> +	for (next = start; next < end; next = range.end) {
> +		range.start = next;
> +		range.end = min(end, next + (PFNS_ARRAY_SIZE << PAGE_SHIFT));
> +		...
> +		ret = do_fault_and_migrate(dmirror, &range);

Inside do_fault_and_migrate(), migrate_hmm_range_setup() computes:

    migrate->npages = (migrate->end - migrate->start) >> PAGE_SHIFT;

With start=0 and end=0 this is always 0.  The for-loop in
migrate_hmm_range_setup() never runs, migrate->cpages stays 0, and
then migrate_vma_pages() and migrate_vma_finalize() both iterate over
0 entries in __migrate_device_pages() / __migrate_device_finalize()
and do nothing.

This means any migration entries installed inside hmm_range_fault()
are never removed.  Subsequent accesses to those virtual addresses
will block on migration_entry_wait() indefinitely.

Compare with dmirror_migrate_to_device(), which sets args.start and
args.end explicitly on every loop iteration before calling
migrate_vma_setup().  The new path skips that.

Either migrate_hmm_range_setup() should derive npages from
(range->end - range->start) >> PAGE_SHIFT rather than from
migrate->end - migrate->start, or the caller must set migrate.start
and migrate.end inside the loop.

Also worth noting: do_fault_and_migrate() calls all four migration
functions regardless of what dmirror_range_fault() returned.  If the
range is invalidated and -EBUSY comes back with migration entries only
partially installed, the cleanup chain still needs to run.
